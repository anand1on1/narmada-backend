# R27.1a — Deploy Guide

## Overview
R27.1a is an additive bugfix sweep on top of R27.1. Backend deploys to Render; the frontend ships as a static zip to GoDaddy.

---

## 1. Backend (Render — narmada-backend.onrender.com)
1. Merge / push `main` (this commit). Render auto-builds from `main`.
2. On boot, confirm these log lines:
   ```
   [boot] brevo configured: yes        ← must be "yes" in prod (env keys set)
   [migrations] R27.1a: shop_users.email_verified added   (or "skip" on re-deploy)
   [migrations] R27.1a: ... verify_otp / verify_otp_expires_at / verify_otp_sent_at
   [migrations] R27.1a: grandfathered N pre-existing shop_users as verified
   [migrations] R27.1a: complete
   [boot] listening on PORT ...
   ```
3. The migration is idempotent — re-deploys log `skip (duplicate column)` and are safe.
4. **No manual DB step required.** Columns are added automatically; no table is dropped or renamed. SQLite lives at `/opt/render/project/src/data`.

### Required env (already set in prod — verify present)
- `BREVO_SMTP_HOST`, `BREVO_SMTP_USER`, `BREVO_SMTP_KEY` (or PASS), `EMAIL_FROM` — without these, OTP/welcome/status emails will not send (login still gates correctly, but users can't receive codes).

## 2. Frontend (GoDaddy — static)
1. Build artifact already produced: `/home/user/workspace/narmada-mobility-godaddy.zip`.
2. `client/public/config.js` points the SPA at `https://narmada-backend.onrender.com` (verify after upload).
3. Upload the zip contents to the GoDaddy web root. Hash routing means `/#/customer/verify`, `/#/admin/web-customers/:id`, etc. all resolve to `index.html` — no server rewrite rules needed.

## 3. Post-deploy smoke (see R27.1a-TEST-CHECKLIST.md)
- New shop signup → OTP email lands → verify → auto-login.
- Existing R27.1 customer can still log in (grandfathered).
- Admin → Web Customers → click a row → detail (orders/addresses/wishlist).
- Admin PO list: search + date filter; Process PO; Duplicate PO; Duplicate Quotation.
- Sidebar shows **Web Orders** between Quotes and Purchase Orders.

## Rollback
- Frontend: re-upload the previous R27.1 zip.
- Backend: redeploy the previous commit. The added columns are harmless if unused; no data migration to reverse.
