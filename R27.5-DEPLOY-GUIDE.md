# R27.5 — Deploy Guide

Two halves to deploy: **backend** (Render) and **frontend** (GoDaddy static zip). Backend first so the new endpoints exist before the new UI calls them.

## 1. Backend → Render
1. Push `main` (this commit) to the repo Render builds from.
2. Render runs `npm run build` and starts `node dist/index.cjs`.
3. On boot, `runR27_5Migrations()` runs automatically against the mounted `data.db` (DATA_DIR). It is **additive and idempotent** — safe to run on the live DB. Watch the logs for:
   ```
   [migrations] R27.5: start
   [migrations] R27.5: … added / ok        (one line per column/table)
   [migrations] R27.5: complete
   [boot] listening on PORT …
   ```
   Any `skip (duplicate column name)` lines are normal on re-deploy (column already exists).
4. No env-var changes required.

## 2. Frontend → GoDaddy
1. Download `narmada-mobility-godaddy.zip` from the workspace root.
2. In GoDaddy cPanel File Manager, upload and extract into the web root (replacing the previous build). The zip already contains `.htaccess` (SPA rewrite + cache headers) and `config.js`.
3. Confirm `config.js` reads:
   ```js
   window.__API_BASE__ = "https://narmada-backend.onrender.com";
   ```
   (already set in the zip — only change if the backend URL differs).
4. Hard-refresh (Ctrl/Cmd-Shift-R) to bust cache.

## 3. Smoke test after deploy
- Public site: header does not overlap the wordmark; currency badge is the compact pill; switch INR/USD.
- Add a product to cart → Checkout → COD → order places without "Failed to fetch".
- Admin → Stock: Live (net) and Detailed toggle both load.
- Admin → Freight Charges: search "patna" returns rows; add one with Source/Destination/Mode.
- Admin sidebar: groups collapse/expand, search filters, state survives reload.
- Finance portal → Accounts: Employees (search + new), Cash (branch in/out), Current Expenses (approve a >₹5000 one as admin), Person Ledger.

## Rollback
- Frontend: re-upload the previous zip.
- Backend: redeploy the prior commit. R27.5 migrations are additive, so the older server simply ignores the new columns/tables — no data loss, no down-migration needed.
